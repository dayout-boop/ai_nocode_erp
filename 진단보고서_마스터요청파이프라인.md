## (A) RAG 조회 진단 — "10건"의 정체와 "2096건"의 출처

### 사실
- 마스터 채팅(`/erp/master-ai`)은 `POST /api/master-stream`(`server/masterStream.ts`)을 사용한다.
- 의도분류(`classifyIntent`, `server/services/rag.ts`)가 예약/정산/통계 키워드를 감지하면 `needsReservations`가 켜지고, **`fetchReservationContext`가 호출된다.**
- 그런데 `fetchReservationContext`는 이름과 달리 **`bookings` 테이블 최근 10건만 `LIMIT 10`** 으로 조회한다(고정 코드). 입력 텍스트의 명확/불명확과 무관하게 항상 같은 10건이 주입된다.
- 자금 정본인 **`income_records`(자금 5종 정본)** 와 **`reservations`(구 예약 정본)** 는 RAG 기본 컨텍스트에서 **전혀 조회되지 않는다.**
- 마스터 Tool Calling 도구 16종(`server/services/masterTools.ts`)을 전수 확인한 결과:
  - `get_bookings_summary` → `bookings`
  - `get_settlement_summary` → `settlements`
  - `get_system_health` → bookings/aiCostLogs/aiEngineLogs/devRequests/inquiries
  - 그 외 packages/inquiries/users/aiLogs/devRequests 등
  - **`income_records` 또는 `reservations`의 unmatched 건수를 세는 도구가 존재하지 않는다.**

### 결론
- 사용자가 의심한 **"이름은 정산인데 다른 DB(bookings)를 본 것"이 맞다.**
- 요청 본문의 **"unmatched income 2096건" 수치는 DB 조회로 산출된 것이 아니라**, 이전 세션/이전 보고값이 대화 컨텍스트에 남아 LLM이 **재인용**한 것으로 판단된다. (RAG도 도구도 income unmatched를 세지 않으므로 그 외 경로가 없음)
- **권한 문제는 아니다.** adminProcedure·도구 권한 모두 통과한다. 단지 **자금 정본 조회 도구가 미구현**일 뿐이다.

### 권고
1. `masterTools.ts`에 **`get_income_match_summary`** 도구 신설: `income_records`에서 `status='unmatched'` 카운트, tenantId별 집계, 매칭 후보 예약 수 등 반환.
2. `fetchReservationContext`를 정정: bookings 외에 정산/자금 질의 시 `income_records` 미매칭 요약을 함께 주입하거나, 함수명을 실제 대상(bookings)에 맞게 명확화.
3. "사실 기반 응답" 원칙(master.ts L16~20)과 정합: 도구로 세지 못하는 수치는 인용하지 않도록 가드.

## (B) 마스터 대화 이력 "안 보임" — 데이터 유실 아님

### 사실
- `ai_logs` 테이블 행 수 = **813행** (DB 직접 확인). 데이터 정상.
- `listMasterSessions`(`server/routers/chat.ts`)는 `assistant='master'` 로그를 sessionId별로 그룹핑해 목록화한다. 화면 "마스터 대화 이력 (11개 세션)"에 6/6~6/10 기록이 모두 표시됨.
- `getMasterSessionMessages`로 세션별 전체 메시지 조회 가능. `loadSessionHistory`(이어가기용)도 이미 구현됨.

### 결론
- **데이터는 사라지지 않았다.** "두골프 마스터" 채팅창은 **현재 단일 세션만** 렌더링하고, 과거 세션은 별도 "마스터 대화 이력" 페이지(`/erp/master-ai/logs`)에서 세션별로 본다. 채팅창과 이력 목록이 **분리된 UI 구조**가 "안 보인다"의 원인.

### 권고 (택1)
1. 채팅 진입 시 **최근 세션 자동 로드**(loadSessionHistory 활용).
2. 채팅 화면에 **세션 전환 드롭다운/사이드 목록** 추가.
3. 현행 유지(이력은 logs 페이지에서만 열람).

## (C) 요청 의도 보존 — 변질 방지 강제 장치 부재

### 사실
- 마스터 시스템 프롬프트(`server/services/prompts/master.ts` L77~92)의 dev_request 생성 지침은 `title/description/priority/module/estimatedHours`만 요구하며, **"사용자 원문 보존" 규칙이 없다.** description은 LLM이 자유 재작성.
- dev_request 감지 후 `devRequest.classifyRequest` → `autoRegisterAndSend`에서 **LLM이 한 번 더 title/description/module/우선순위를 재가공**한다(2단계 LLM 가공).
- 따라서 신규문서 작성·마스터 기능 요구·업체 규율/상위규정·테넌트 분리·AI 로직·부킹 등 **인접 주제로 범위가 번질(scope creep) 구조적 여지**가 있다.

### 이번 660001 요청 점검
- 제목/설명/AI분석 모두 "자금 일괄 대사 + tenantId 상속 + unmatched income"으로 일관 → **이번엔 의도가 보존됨.**
- 단, 이는 강제 규칙 덕분이 아니라 **운에 가깝다.** 변질을 막는 장치가 없다.

### 권고
1. master.ts에 **"사용자 원문 발췌(verbatim) 필드 보존"** 규칙 추가: dev_request JSON에 `originalRequest`(사용자 발화 원문) 필드를 의무화하고, 마누스 전송 메시지에 원문 블록을 별도 동봉.
2. `classifyRequest`의 LLM 재가공은 **분류(priority/module/hours)에 한정**하고 title/description은 마스터 1차 생성본을 보존하도록 분리.

## (D) 타 데스크 지식 차단 — 규칙만 존재, 자동 적용 없음

### 사실
- `server/services/knowledgeFilter.ts`에 차단 규칙 **12종**(GitHub 연동/IP 보호/L-5 인가/파일수정 3회 중단/이데스크 운영/최고규율/OpenManus/내부정보/다중환경/자율수행 전략/외부리소스 제한/연결문제 분석)이 키워드 기반으로 정의됨. 화면 "기본 차단 규칙 12개"와 일치.
- 그러나 `checkKnowledge`/`filterKnowledgeList`를 호출하는 곳은 **`knowledgeBlock.checkKnowledge` 프로시저 한 곳뿐**이며, 이는 **"마스터가 직접 입력해 테스트"하는 수동 검사용**이다(주석 명시).
- 마스터 채팅(`masterStream`)·개발요청(`manusPipe`/`devContext`) 파이프라인에서 **자동 호출되지 않는다.** 화면 **"총 차단 건수 0"** 이 무자동작을 입증.

### 핵심 한계
- 이 필터가 검사할 수 있는 대상은 **ERP DB에 저장되는 텍스트**뿐이다.
- 정작 중요한 **Manus 플랫폼이 에이전트에게 주입하는 `related_knowledge`(타 데스크 지식)** 는 **Manus 레이어에서 발생**하므로 **ERP 서버 코드로는 가로채거나 차단할 수 없다.** (구조적 불가)
- 따라서 그 차단은 **프로젝트 지침(projectInstruction) + 에이전트의 지침 준수**로만 보장된다. 본 진단 작업 중에도 타 데스크 지식이 실제 주입되었으나, 지침에 따라 적용하지 않았다.

### 권고
1. ERP 내부 차단 필터를 **실제 파이프라인에 연결**: masterStream/manusPipe 입력 단계에서 `filterKnowledgeList`를 자동 호출해 ERP DB에 유입되는 타 데스크 텍스트를 걸러내고 로그 기록(최소한 ERP 측 방어선 확보).
2. "Manus 주입 지식 차단"은 ERP로 불가함을 명확히 인지하고, **projectInstruction의 SYSTEM OVERRIDE 강제 명령에 의존**하는 현 구조를 공식 정책으로 문서화.
3. 차단 UI에 "이 필터는 ERP DB 유입 텍스트 대상이며, Manus 플랫폼 주입 지식은 지침으로 통제됨"을 명시해 오해 방지.
